# Discipline-Aware Skill Factory

> **Execution:** Use `/dev-workflow:execute-plan docs/exec-plans/active/discipline-aware-skill-factory.md` to implement milestone-by-milestone.

## Purpose / Big Picture

Transform the `dev-skill` plugin from a single-discipline skill generator (Library/API Reference only) into a discipline-aware factory that can produce 9 types of skills across 4 production disciplines. After this change, users run `/dev-skill:new`, pick a skill type from a menu, and get a generation workflow tailored to that type — not a one-size-fits-all "40 rules" pipeline.

The Anthropic blog post on skills (March 2027) identifies 9 skill types used at scale. The current plugin handles 2 of them (Library/API Reference, Code Quality). This evolution covers all 9.

### Terms

| Term | Meaning |
|------|---------|
| **Discipline** | A production method for creating a skill's core substance. Four disciplines: Distillation, Composition, Extraction, Investigation. |
| **Distillation** | Take authoritative sources and distill into progressive-disclosure reference rules. Current plugin does this. |
| **Composition** | Compose tools/MCPs/scripts into workflow instructions. For verification, automation, CI/CD, infra ops skills. |
| **Extraction** | Analyze codebases and derive templates/conventions. For scaffolding skills. |
| **Investigation** | Build diagnostic decision trees and query patterns. For runbook and data analysis skills. |
| **Anatomy** | Universal skill structure all disciplines share: SKILL.md format, gotchas pattern, setup/config pattern, progressive disclosure. |
| **Recipe** | Per-discipline document describing how to produce a skill of that type. Replaces the single MENTAL_MODEL.md. |
| **Rubric** | Per-discipline validation checklist with verifiable (not subjective) checks. Fed to the skill-reviewer agent. |
| **Bootstrap** | Initial generation of a skill. Depth varies by discipline (full for Distillation, starter for others). |
| **Evolve** | Type-aware improvement of an existing skill. Replaces `/improve`. |

3. Write `RUBRIC.md` — verifiable substance checks for the skill-reviewer:
   ```markdown
   # Distillation Validation Rubric

   ## Substance Checks (reviewer agent follows these)

   ### Claim Verification (pick 3 random rules)
   For each selected rule:
   1. WebSearch the core claim. Does authoritative documentation agree?
   2. Run the "incorrect" example mentally — would it actually produce the stated problem?
   3. Does the "correct" example compile/lint in the stated technology?
   4. Is the impact level calibrated correctly relative to sibling rules?
   5. Is the impactDescription quantified, not vague?

   ### Cross-Rule Consistency
   1. Are there rules that contradict each other?
   2. Are there rules that duplicate each other (same advice, different wording)?
   3. Is the highest-impact category genuinely the most impactful?
   4. Are CRITICAL rules actually critical, or is impact inflated?

   ### Reference Verification (spot-check 3-5 URLs)
   1. Are they still live?
   2. Do they support the claim the rule makes?
   3. Are they from authoritative sources (official docs, framework maintainers)?

   ### Completeness
   1. Are major performance concerns for this technology covered?
   2. Are there obvious patterns missing that appear in 3+ authoritative sources?
   3. Is the "When NOT to use" section present for rules with important exceptions?

   ### Verdict Criteria
   - SHIP: All 3 sampled rules pass claim verification. No contradictions. References live.
   - NEEDS WORK: 1-2 sampled rules fail claim verification, or minor contradictions found.
   - REJECT: Systematic inaccuracy, or majority of sampled rules fail verification.
   ```

4. Move template files from `templates/skill-generator/` to `templates/disciplines/distillation/`. Keep copies in the old location during transition (symlinks or just leave both).

5. Delete `templates/skill-generator/MENTAL_MODEL.md` — its content is now split between `anatomy/ANATOMY.md` and `disciplines/distillation/RECIPE.md`.

### Milestone 3: Composition discipline

**Goal:** Build the Composition discipline for Verification, Automation, CI/CD, and Infrastructure Ops skills. These skills produce scripts, hooks, and workflow instructions rather than reference rules.

**Work:**

1. Create `templates/disciplines/composition/`:
   ```
   templates/disciplines/composition/
     RECIPE.md               # Composition methodology
     RUBRIC.md               # Verifiable validation checks
     SKILL.md.template       # Composition-specific entry point
     workflow.md.template    # Workflow step template
     script.sh.template      # Script scaffold template
     hooks.json.template     # On-demand hook template
     config.json.template    # Setup config template
   ```

2. Write `RECIPE.md` — the composition production methodology:
   - **Inputs needed:** Tool/MCP inventory, workflow description, success criteria, risk assessment
   - **Discovery workflow:**
     1. Ask user: What process are you automating? What tools do you use? What does success look like?
     2. Map the workflow steps: trigger → steps → verification → cleanup
     3. Identify tool dependencies (MCPs, CLI tools, APIs)
     4. Identify risk level: read-only / write / destructive
   - **Output structure:**
     ```
     {skill-name}/
     ├── SKILL.md              # Workflow description + trigger phrases
     ├── metadata.json         # Includes discipline: "composition", type field
     ├── config.json           # User-specific setup (channels, paths, credentials refs)
     ├── gotchas.md            # Failure points discovered over time
     ├── scripts/              # Executable scripts the skill uses
     │   ├── {step-name}.sh    # Individual workflow steps
     │   └── verify.sh         # Verification/assertion script
     ├── hooks/                # On-demand hooks (active during skill session)
     │   └── hooks.json        # Hook definitions
     └── references/
         └── workflow.md       # Detailed workflow documentation
     ```
   - **Generation pipeline:**
     1. User interview (AskUserQuestion) → tool inventory + workflow steps
     2. Planning checkpoint (display workflow diagram, ask approval)
     3. Generate SKILL.md with trigger phrases and workflow overview
     4. Generate scripts for each workflow step (with error handling)
     5. Generate verification script (assertions for success criteria)
     6. Generate hooks if applicable (e.g., PreToolUse guardrails for destructive ops)
     7. Generate config.json template for user-specific values
     8. Validate with composition rubric
   - **Risk-level handling:**
     - Read-only (data fetching, analysis): No guardrails needed
     - Write (creating PRs, posting messages): Confirmation before external side effects
     - Destructive (delete resources, force-push): PreToolUse hook blocks + explicit confirmation
   - **Type-specific guidance:**
     - Verification skills: Focus on assertions, recording, state checking. Include a `scripts/verify.sh` that can programmatically assert state.
     - Automation skills: Focus on workflow composition, logging, idempotency. Save results in log files for cross-run consistency.
     - CI/CD skills: Focus on multi-step pipelines with rollback. Each step must have a rollback counterpart.
     - Infra Ops skills: Focus on guardrails. Must include dry-run mode. Destructive actions require soak period + confirmation.

3. Write `RUBRIC.md` — verifiable checks for composition skills:
   ```markdown
   # Composition Validation Rubric

   ## Substance Checks

   ### Script Validity
   1. For each script in scripts/: does `bash -n {script}` pass syntax check?
   2. Are all referenced tools/commands available? (check with `command -v` or `which`)
   3. Do scripts use `set -euo pipefail` for safety?
   4. Are error messages actionable (tell user what to do, not just what failed)?

   ### Workflow Completeness
   1. Does every workflow step have a clear trigger and success condition?
   2. Are failure paths handled? (What happens when step 3 fails?)
   3. Is there a verification step that asserts the workflow succeeded?
   4. For multi-step workflows: can the workflow be resumed from a failed step?

   ### Guardrail Assessment (for write/destructive skills)
   1. Are destructive operations guarded by confirmation prompts?
   2. Is there a dry-run mode?
   3. Do guardrail hooks actually block the dangerous operations?
   4. Are rollback instructions documented for each destructive step?

   ### Setup & Config
   1. Does config.json have clear descriptions for each field?
   2. Does the skill gracefully handle missing config (ask user, don't crash)?
   3. Are secrets referenced by environment variable, never hardcoded?

   ### Usefulness
   1. Does this workflow actually save time vs doing it manually?
   2. Is the workflow generic enough to work across different projects?
   3. Would an engineer trust this to run without supervision?

   ### Verdict Criteria
   - SHIP: Scripts pass syntax check. Error paths handled. Guardrails present for destructive ops.
   - NEEDS WORK: Scripts valid but missing error handling or guardrails.
   - REJECT: Scripts don't execute. Destructive ops ungated. Workflow has dead ends.
   ```

4. Write template files:
   - `SKILL.md.template` — Composition-specific with workflow overview section
   - `workflow.md.template` — Step-by-step workflow with trigger/action/verify/rollback per step
   - `script.sh.template` — Safe script scaffold with error handling
   - `hooks.json.template` — On-demand hook template for guardrails
   - `config.json.template` — Setup config with field descriptions

**Result:** Composition discipline is ready. Covers 4 skill types (Verification, Automation, CI/CD, Infra Ops).

**Proof:** All template files exist and are self-consistent. `RECIPE.md` covers all 4 type variants. `RUBRIC.md` has verifiable checks (not subjective).

## Idempotence and Recovery

Each milestone is independently committable. If any milestone fails:
- Milestones 1-5 are purely additive (new files). Rollback: delete the new files.
- Milestone 6 modifies commands. Rollback: `git checkout -- plugins/dev-skill/commands/`
- Milestone 7 modifies agents/scripts. Rollback: `git checkout -- plugins/dev-skill/agents/ plugins/dev-skill/scripts/ plugins/dev-skill/hooks/`
- Milestone 8 is cleanup. Rollback: `git checkout -- plugins/dev-skill/`

The safest order (additive first, destructive last) is already the planned order. Old templates aren't deleted until Milestone 8 confirms everything works.
